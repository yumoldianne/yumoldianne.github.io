Why your A/B test is probably wrong

A practical look at the assumptions we violate every day and what to do about it.

January 10, 2025
·
8 min read
·
Statistics Experimentation

A/B testing is the gold standard of causal inference in product teams — or so the story goes. Run an experiment, flip a coin, measure a metric, ship the winner. Simple. Except it's almost never that clean, and the assumptions hiding beneath the surface fail more often than most practitioners want to admit.

TL;DR: Most A/B tests violate at least one of SUTVA, randomization integrity, or multiple testing assumptions. This post walks through each, with practical fixes.

The problem with "just run a test"

The canonical A/B test assumes two things: that units don't interfere with each other (SUTVA), and that the randomization actually produced comparable groups. In practice, both are routinely violated — and the damage is often invisible.

Consider a recommendation system test. You assign users to treatment or control at random. But if treatment users start sharing links that control users click, you've just introduced interference. Your control group is contaminated. Your effect estimate is biased. And your p-value means nothing.

SUTVA violations

The Stable Unit Treatment Value Assumption (SUTVA) has two parts: no interference between units, and no hidden versions of treatment. Violating either one collapses the neat potential outcomes framework that justifies the whole procedure.

Network interference

Network interference is the most common violation in consumer apps. Social features, viral loops, shared inventory — anything that creates interdependence between users will bleed treatment into control. The standard fix is cluster-based randomization: assign entire clusters (friend groups, geographic regions, device families) rather than individuals. 

# Cluster-based randomization in Python
import numpy as np
import pandas as pd

def cluster_randomize(df, cluster_col, treatment_share=0.5, seed=42):
    rng = np.random.default_rng(seed)
    clusters = df[cluster_col].unique()
    n_treat = int(len(clusters) * treatment_share)
    treated = rng.choice(clusters, size=n_treat, replace=False)
    df['treatment'] = df[cluster_col].isin(treated).astype(int)
    return df

Hidden versions of treatment

The second part of SUTVA — no hidden versions — is violated whenever "treatment" isn't uniformly delivered. A/B frameworks that rely on client-side feature flags can serve different versions of the same variant depending on device type, network latency, or SDK version. The result: your "treatment" is actually several different treatments averaged together.

The multiple testing trap

If you run 20 tests simultaneously and use α = 0.05, you expect one false positive by chance alone — even if every null is true. Most teams implicitly run dozens of experiments at once across their product, then congratulate themselves on "data-driven" decisions without any correction for familywise error rate.

Practical fix: Use Benjamini–Hochberg correction for simultaneous hypotheses, or adopt a sequential testing framework (e.g. always-valid p-values) that lets you peek without inflating error rates.

What to actually do

There's no single silver bullet, but a few practices substantially reduce the risk of drawing wrong conclusions from your experiments:

  • Pre-register your primary metric and sample size calculation before running
  • Use cluster randomization when units interact socially or geographically
  • Apply multiple testing correction if you track more than one metric
  • Run A/A tests periodically to validate your randomization pipeline
  • Document every decision made mid-experiment — that's where p-hacking lives

None of this makes A/B testing "wrong" as a tool — it remains the cleanest way to establish causality at scale. The point is to go in with eyes open about what the method can and can't do. The confidence interval is only as good as the assumptions underneath it.

"All models are wrong, but some are useful." The same applies to experiments.
