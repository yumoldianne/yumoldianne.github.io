Making core genome phylogenies is hard (but we tried to make it easier)

A Nextflow pipeline that filters bacterial core genes by how informative they actually are — testing whether a smaller, smarter gene set can build a phylogenetic tree that's just as good, for a fraction of the compute.

Team project · with Marc A. & RJ C.
·
8 min read
·
Nextflow Bioinformatics HPC / SLURM Test-Driven Dev
1,529 core genes · 299 M. tuberculosis genomes · 1 filtering pipeline

Phylogenomics — building evolutionary trees from whole-genome data instead of a single gene — is how researchers trace tuberculosis outbreaks, track drug-resistant lineages, and figure out whether a strain jumped from humans to animals. The problem is that "whole-genome" analysis on real bacterial datasets means aligning and running maximum-likelihood inference on thousands of genes across hundreds of genomes, which can mean days of runtime and dozens of GB of memory for a single tree. Our internship project asked a simple question with a not-so-simple answer: do you actually need all of that data to get a good tree?

TL;DR: We built a test-driven Nextflow pipeline that filters bacterial core genes by three informativeness metrics before tree-building, then benchmarked 42 configurations against a M. tuberculosis dataset to see whether filtering saves compute without sacrificing tree quality. The honest answer: filtering saved real compute time, but every configuration we tested produced a less accurate tree than using all the genes — the pipeline works, but "filter aggressively" turned out to be the wrong lesson to take from it.

The question: are all core genes equally informative?

"Core genes" are the genes shared across every genome in a dataset — the common substrate any phylogenomic tree gets built from. The assumption behind most pipelines is that you use all of them. Our hypothesis was that some core genes carry a lot more phylogenetic signal than others, and that scoring and filtering down to just the informative subset (V ⊂ U, where U is all core genes and V is the informative ones) might produce a tree that's just as good — or nearly as good — for a fraction of the computational cost.

Approach: three filters, three phylogeny tools, one pipeline

We scored each core gene alignment on three metrics before deciding whether to keep it:

  • Number of polymorphic sites — a raw count of variable positions, a basic measure of genetic variation.
  • Nucleotide diversity — a normalized measure of genetic variation across the alignment.
  • dN/dS ratio — the ratio of non-synonymous to synonymous substitutions, a standard measure of the selective pressure acting on a gene.

Filtered gene alignments were concatenated and run through a maximum-likelihood tree-building step (with a choice of RAxML-NG, IQ-TREE2, or FastTree), and the resulting trees were scored against a reference tree built from the complete, unfiltered gene set.

The engineering: building it like software, not a one-off script

A detail that mattered more than expected: this wasn't built as a linear analysis script, it was built as a Nextflow pipeline using test-driven development — write a failing test for a feature, write the minimum code to pass it, then refactor. Each feature request mapped to a specific pipeline stage (filtering, phylogeny construction, or measuring), which meant every filter and every tree-building option could be validated in isolation before being wired into the full pipeline.

That paid off directly: 18/18 unit tests and 14/14 integration tests passed on the full pipeline. It also meant that when two components hit real scaling problems — RAxML-NG took over three days on the full dataset even with 24 CPUs and 24 GB of memory, and the dN/dS filtering step took roughly a day on its own — we caught it before it silently corrupted the final benchmark, and could simply exclude those specific configurations from data collection rather than losing the whole run.

Production features included submitting individual modules as SLURM jobs on the university's HPC cluster, filtering gene alignments independently in parallel, Docker/Singularity containerization for reproducibility, and pipeline checkpoints so a multi-day HPC job could resume instead of restarting from scratch after a failure.

The dataset: 1,529 core genes, 299 curated genomes

We worked from an existing M. tuberculosis dataset of ~23,500 gene alignments across 1,730 sequences. From that, 1,529 genes qualified as true core genes (present across the full set), and we interactively selected 299 representative accession IDs using iTOL — balancing taxonomic breadth against the compute budget available on the shared HPC cluster. The final working dataset, labeled ghana0, was 299 genomes × 1,529 core gene alignments.

The benchmark: 42 pipeline configurations, five metrics each

With the pipeline validated, we ran 42 configurations sweeping polymorphic-site cutoffs, nucleotide-diversity ranges, dN/dS ranges, and phylogeny-building tool (FastTree vs. IQ-TREE2), capturing five outcomes per run: filtered gene count, Robinson-Foulds (RF) distance to the unfiltered reference tree, average bootstrap support, CPU time, and peak memory.

Because this was a single dataset with no biological or technical replicates, we deliberately avoided ANOVA or t-tests — both require replicate-based variance estimates we didn't have. Instead we used non-parametric tests (Mann-Whitney U, Kolmogorov-Smirnov) to compare distributions across tools, and simple/multiple regression to quantify which factors actually drove each outcome.

What we found: filtering costs accuracy to buy speed

The cleanest result was on compute cost: phylogeny tool choice dominates resource usage (R² = 0.96 for both CPU time and memory), with FastTree consistently far cheaper than IQ-TREE2 across every filtering configuration we tested. That part matched expectations — different tree-building algorithms really do have very different cost profiles.

The less comfortable result was on quality. Stricter filtering (lower polymorphic-site cutoffs, tighter nucleotide-diversity ranges) consistently increased RF distance from the reference tree and decreased average bootstrap support — meaning the filtered trees were both less similar to the "ground truth" complete-gene tree and less internally confident about their own branching. poly_cutoff and nucdiv_start were the strongest, most consistent predictors of tree quality across every model we ran, and the effect ran in the wrong direction for our hypothesis: filtering more aggressively didn't isolate a more informative signal, it just threw away signal.

Honest result, not a positive one: every filtered configuration we tested produced a tree of lower quality than the complete-gene tree. The pipeline itself worked exactly as designed — it's the underlying hypothesis (aggressive filtering preserves tree quality) that the data didn't support at the cutoff values we tried.

Why this matters beyond one internship dataset

This project's value isn't really "we proved filtering works" — it's a working piece of infrastructure plus a clear, evidence-backed answer about where its limits are, which is arguably more useful to a research group than a positive result would have been:

  • A reusable tool, not a one-off analysis. The pipeline is parameterized, containerized, SLURM-aware, and checkpointed — a lab studying a different pathogen can point it at a new dataset and sweep their own filtering constraints without touching the underlying code.
  • A negative result that saves other people time. Anyone tempted to speed up phylogenomic analysis by filtering on polymorphic sites or nucleotide diversity now has quantified evidence of the accuracy cost at various cutoff levels, instead of having to discover it themselves after days of compute.
  • The real lever we found was tool choice, not gene filtering. Since phylogeny method — not filtering — explained 96% of the variance in compute cost, the actionable takeaway for a lab with limited HPC time is "pick FastTree over IQ-TREE2 when appropriate," not "filter your core genes," which is a more useful and lower-risk recommendation than the one we set out to test.
  • Test-driven development made a negative result trustworthy. Because every pipeline stage had unit and integration tests before we ever touched the real dataset, we could be confident the "filtering hurts quality" finding reflects real biology and not a pipeline bug — which matters a lot when the result contradicts your starting hypothesis.

Limitations

Several caveats limit how far these findings generalize. The 299-genome M. tuberculosis dataset may have been too homogeneous — a famously clonal, low-diversity pathogen — for filtering to show its best-case benefit; a more genetically diverse organism might respond differently. We tested only relatively strict cutoff ranges and didn't explore whether much gentler filtering might preserve quality while still trimming meaningful compute. We also had no true "random filtering" control to isolate the effect of principled filtering from just having fewer genes overall, and with a single dataset and no replicates, our statistical power is inherently limited — these results may be specific to this pathogen and this gene set rather than a general property of core-gene filtering.

Full pipeline code, benchmark configurations, and the complete regression output are available at github.com/neotechz/core-phylogenies.
