Context-Aware Residual Blocks: teaching a network to discover its own simplicity

A multi-path residual block with learned, input-dependent routing — and what happened when I let a neural network choose how complicated it wanted to be.

CSCI 214 Final Paper
·
9 min read
·
Deep Learning PyTorch Architecture Design Unsupervised Learning
Identity · Linear · Non-linear
three paths, one learned router

Every residual network since ResNet has made the same bet: add a skip connection, always use it, always weight it 50/50 with the learned transformation. That fixed 1:1 split is what lets very deep networks train — but it also means the network has no say in how much "shortcut" versus "transformation" any given input actually needs. I wanted to know what happens if you let the network decide.

TL;DR: I built Context-Aware Residual Blocks (CARB) — three parallel paths (identity, linear, non-linear) combined through a tiny learned router that reads batch statistics, training progress, and layer depth. On Fashion-MNIST, CARB barely helped supervised accuracy (+0.19 points) but transformed unsupervised clustering (+95.9% silhouette score) — and along the way it taught itself that Fashion-MNIST barely needs the non-linear path at all.

The idea: routing instead of a fixed shortcut

Standard ResNet computes y = F(x) + x. CARB replaces this with a weighted sum over three parallel branches: 

y = w1(c)·P1(x) + w2(c)·P2(x) + w3(c)·P3(x)

P1(x): identity / learned projection      — a pure shortcut
P2(x): BatchNorm(W2x + b2)                — one linear layer
P3(x): two-layer bottleneck + ReLU        — the "expensive" nonlinear path

[w1, w2, w3] = softmax( MLP(context) )
context = [batch mean, batch std, training progress, layer depth]

The router is deliberately tiny — about 163 parameters per block — and it never sees the raw input directly. It only sees aggregate statistics about the current batch and where the block sits in training and depth. That's the whole trick: a cheap, differentiable dispatcher deciding, per batch and per layer, how much of each computational "flavor" to use.

This is a genuinely architectural change, not a fancier activation function. All three paths have to be computed in parallel inside the same forward pass, Path 3 is structurally deeper than Paths 1 and 2, and the routing weights depend on aggregate batch statistics rather than acting element-wise. None of that can be reproduced by swapping in a different nonlinearity.

Experimental setup

I evaluated CARB on Fashion-MNIST in two very different regimes, against a plain feed-forward baseline and a standard fixed-weight ResNet:

  • Supervised classification — three architectures (baseline FFNN, standard ResNet, CARB network), all trained with Adam, cosine annealing, batch size 128, 30 epochs.
  • Unsupervised representation learning — symmetric autoencoders (vanilla vs. CARB-based encoder/decoder), evaluated by K-Means silhouette score on the 32-dimensional latent space.

CARB edges out the baseline by 0.19 points and beats standard ResNet by 0.45 — but it costs 81% more parameters to get there, roughly a million extra parameters per percentage point of accuracy. Standard ResNet, meanwhile, actually does worse than the plain baseline despite having more capacity, which was its own useful finding: adding skip connections without any routing mechanism isn't free — it can hurt.

Per-class analysis told a more interesting story than the headline number. CARB reallocates representational capacity rather than uniformly improving everything: it picks up +2.7 points on "Shirt" and +2.9 on "Pullover" — Fashion-MNIST's most visually confusable categories — while giving up ground on easier classes like T-shirt/top (-1.8) and Dress (-1.1). And CARB overfits harder than either baseline: 99.5% training accuracy against a validation accuracy of only 90.14%, a 9.4-point train/validation gap versus the baseline's 3.0 points.

Unsupervised results: where the architecture actually shines

This is the result I didn't expect going in. A standard 32-dimensional autoencoder actually destroys cluster structure relative to raw pixels — its bottleneck concentrates information into 8–12 dominant dimensions and treats the rest as noise (effective dimensionality ≈ 10.2 out of 32). The CARB autoencoder, using the exact same 32-dimensional bottleneck, spreads information across nearly all of it (effective dimensionality ≈ 22.7), producing tighter clusters (43% lower intra-cluster distance) and better-separated ones (31% higher inter-cluster distance).

Why this matters: the multi-path structure keeps the encoder from collapsing to a low-rank representation the way a single fixed nonlinearity does. With no labels and no classification objective anywhere in the loop, architectural flexibility alone recovered — and then dramatically improved on — the structure a vanilla autoencoder threw away.

What the router actually learned

The most surprising finding came from watching the routing weights evolve during training. I had hypothesized a kind of curriculum: heavy use of the non-linear path early (exploration), settling into balance mid-training, and leaning on identity late (exploitation). That's not what happened.

Within the first epoch, both CARB blocks converge hard toward the linear path — by epoch 5, Block 1 is routing 85–90% of its weight through BN(W₂x) alone, and by late training the non-linear path contributes less than 5%. In other words, given the option to use a two-layer nonlinear transformation, the network decided Fashion-MNIST didn't need it and routed around it. Block 2, deeper in the network, kept a larger identity contribution (33–36%) throughout training — consistent with the classic ResNet intuition that gradient highways matter more as you go deeper.

The multi-path structure's real contribution wasn't more capacity — it was giving the network room to discover that it didn't need the capacity in the first place.

Why this matters beyond the benchmark

Fashion-MNIST is a toy problem, but the pattern CARB surfaces is one that shows up constantly in applied ML: teams often reach for deeper or wider architectures by default, without a mechanism to test whether the extra complexity is earning its keep on this dataset. CARB is essentially an architecture that runs that experiment for you, continuously, during training — and reports the answer in its routing weights.

  • Interpretability as a byproduct. Because routing weights are explicit and differentiable, they double as a diagnostic: which inputs are "hard" enough to need the deep path, and which layers are load-bearing for gradient flow. That's a debugging tool, not just an architecture.
  • A concrete efficiency lever. Once weights converge to near-deterministic values (linear ≈0.90–0.95 here), low-weight paths are candidates for post-training pruning — a route back to baseline-level parameter counts without giving up the representational benefit, which I flag as future work rather than something I fully executed.
  • Supervised vs. unsupervised is not the same bet. The 0.19-point supervised gain alone wouldn't justify an 81% parameter increase in a production system. The 95.9% unsupervised gain would — CARB's clearest business case is as a representation-learning or pre-training component (useful wherever labels are scarce or expensive), not as a drop-in classifier upgrade.
  • "Simpler than it needed to be" is itself a finding. A network that's handed three levels of computational complexity and chooses the cheapest one most of the time is good evidence that, for this data, complexity was never the bottleneck — which is exactly the kind of thing a fixed architecture can't tell you.

Limitations

The 191,000 extra parameters for a 0.19-point supervised gain is a genuinely poor cost-benefit ratio at this scale, and the large train/validation gap undercuts any claim that multi-path routing acts as implicit regularization on its own — it likely needs stronger explicit regularization (dropout, weight decay, entropy penalties on the routing distribution) to realize its potential. The near-exclusive preference for the linear path also resembles the routing collapse seen in mixture-of-experts models, and it's unclear whether that pattern would hold on higher-resolution, more complex data where nonlinear transformations plausibly earn their cost.

Full methodology, per-class confusion analysis, and training curves are in the paper. The core takeaway I keep coming back to: giving a model room to be more complex is only useful if you also give it a way to tell you it doesn't want to be.
